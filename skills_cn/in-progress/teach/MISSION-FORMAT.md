# MISSION.md 格式

`MISSION.md` 放在工作区根目录。它记录用户学习该主题的_原因_。每一个教学决策——下一步教什么、展示哪些资源、设计哪些练习——都应追溯到这个文档。

## 模板

```md
# 使命：{主题}

## 为什么
{1-3 句话。用户追求的具体现实目标。当他们拥有这项技能时，他们的生活或工作会发生什么变化？避免抽象表述如"为了理解 X"——追问背后的结果。}

## 成功的样子
- {用户将能做的一件具体、可观察的事}
- {另一件具体的事}
- {……}

## 约束
- {时间、预算、既有承诺、学习偏好、任何限制方法的因素}

## 范围外
- {用户明确不想现在追求的相邻主题——保护最近发展区}
```

## 规则

- **每个工作区一个使命。** 如果用户想学两个不相关的东西，那是两个工作区。
- **具体优于抽象。** "在十月前跑完半程马拉松"胜过"变得更健康"。"给团队发布一个 Rust CLI"胜过"学习 Rust"。
- **对模糊要回推。** 如果用户无法说明原因，在写任何东西之前先采访他们。一个糟糕的使命比没有使命更糟。
- **当现实变化时修订。** 使命会变。当用户的目标移动时，更新这个文件——不要让过时的使命引导未来的会话。
- **保持简短。** 如果 `MISSION.md` 超过一屏，它已经不再是指南针而变成了计划。
