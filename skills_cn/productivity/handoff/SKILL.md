# 工作交接

## 何时交接

- 你需要结束会话但工作尚未完成
- 你正在切换到不同的任务并希望保留上下文
- 你想要创建一个检查点以便稍后回溯
- 工作量很大，你想确保不丢失进度

## 交接过程

1. **评估当前状态** — 回顾本次会话中完成的工作
2. **识别未完成项** — 列出所有待办事项、开放问题和下一步
3. **捕获关键上下文** — 记录做出关键决策的原因，而不仅仅是做了什么
4. **写入交接文档** — 使用下面的模板创建交接文档
5. **保存交接文档** — 将交接文档保存到项目根目录的 `handoffs/` 文件夹中

## 交接文档模板

```markdown
# 工作交接：[任务名称]

## 概要
[1-2 句话描述当前状态]

## 已完成
- [具体已完成的事项]

## 进行中
- [当前正在处理的事项及其状态]

## 待办
- [ ] [下一步事项]
- [ ] [后续事项]

## 关键决策
| 决策 | 原因 | 替代方案 |
|------|------|----------|
| [做出的决策] | [为什么这样做] | [考虑过的其他方案] |

## 开放问题
- [尚未解决的问题]

## 重要上下文
- [其他代理需要知道的关键信息]
- [文件位置、配置、环境详情]
```

## 恢复工作

当在新会话中恢复工作时：

1. 读取 `handoffs/` 目录中最新的交接文档
2. 回顾已完成的工作以避免重复
3. 从进行中/待办事项继续
4. 在进行时更新交接文档

## 最佳实践

- **要具体** — "修改了 auth 模块"不如"在 `src/auth/login.ts` 中添加了 OAuth2 刷新令牌逻辑"
- **记录原因** — "决定使用 Redis"不如"选择 Redis 而不是 Memcached，因为需要持久化以在重启后存活"
- **包含文件路径** — 始终引用被修改或创建的文件的具体路径
- **保持简洁** — 交接文档应该是可快速阅读的，而非详尽无遗的
