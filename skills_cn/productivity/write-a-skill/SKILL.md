# 编写技能

## 过程

1. **收集需求** — 向用户询问：
   - 这个技能覆盖什么任务/领域？
   - 应该处理哪些具体用例？
   - 是否需要可执行脚本还是只需要指令？
   - 有没有要包含的参考资料？

2. **起草技能** — 创建：
   - SKILL.md，包含简洁的指令
   - 如果内容超过500行，创建额外的参考文件
   - 如果需要确定性操作，创建工具脚本

3. **与用户一起审查** — 展示草稿并询问：
   - 是否覆盖了你的用例？
   - 有没有遗漏或不清楚的地方？
   - 是否有部分需要更详细/更简略？

## 技能结构

```
skill-name/
├── SKILL.md           # 主要指令（必需）
├── REFERENCE.md       # 详细文档（如需要）
├── EXAMPLES.md        # 使用示例（如需要）
└── scripts/           # 工具脚本（如需要）
    └── helper.js
```

## SKILL.md 模板

```md
---
name: skill-name
description: 简要描述能力。Use when [具体触发条件].
---

# 技能名称

## 快速开始

[最小可用示例]

## 工作流程

[包含检查清单的复杂任务分步流程]

## 高级功能

[链接到单独的文件：参见 [REFERENCE.md](REFERENCE.md)]
```

## 描述要求

描述是**代理在选择加载哪个技能时唯一能看到的内容**。它会和其他所有已安装技能的描述一起出现在系统提示中。代理会阅读这些描述，并根据用户的请求选择相关的技能。

**目标**：给代理刚好足够的信息，让它知道：

1. 这个技能提供什么能力
2. 何时/为何触发它（具体的关键词、上下文、文件类型）

**格式**：

- 最多1024个字符
- 用第三人称编写
- 第一句话：它做什么
- 第二句话："Use when [具体触发条件]"

**好的示例**：

```
Extract text and tables from PDF files, fill forms, merge documents. Use when working with PDF files or when user mentions PDFs, forms, or document extraction.
```

**坏的示例**：

```
Helps with documents.
```

坏的示例让代理无法将这个技能与其他文档技能区分开来。

## 何时添加脚本

在以下情况下添加工具脚本：

- 操作是确定性的（验证、格式化）
- 相同的代码会被反复生成
- 错误需要显式处理

与生成的代码相比，脚本可以节省令牌（token）并提高可靠性。

## 何时拆分文件

在以下情况下拆分为单独的文件：

- SKILL.md 超过100行
- 内容有截然不同的领域（金融 vs 销售模式）
- 高级功能很少被使用

## 审查清单

起草完成后，验证：

- [ ] 描述包含触发条件（"Use when..."）
- [ ] SKILL.md 在100行以内
- [ ] 没有时间敏感信息
- [ ] 术语使用一致
- [ ] 包含具体示例
- [ ] 引用只有一层深度
