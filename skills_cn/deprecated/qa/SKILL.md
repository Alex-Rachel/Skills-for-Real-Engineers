# 质量保证（QA）

对代码变更执行结构化的质量保证审查。

## 工作流程

### 1. 收集变更

识别需要审查的变更范围：

- 运行 `git diff` 查看已修改的代码
- 识别修改了哪些文件和函数
- 理解变更的意图

### 2. 正确性审查

验证变更是否正确实现了预期行为：

- [ ] 逻辑是否正确？
- [ ] 边界情况是否已处理？
- [ ] 错误处理是否完备？
- [ ] 类型是否一致？
- [ ] 是否存在竞态条件（Race Condition）？

### 3. 安全审查

检查常见安全漏洞：

- [ ] 输入是否已验证/清理？
- [ ] 是否存在注入风险？
- [ ] 敏感数据是否妥善处理？
- [ ] 认证/授权是否正确？
- [ ] 是否存在不安全的依赖？

### 4. 最佳实践审查

对照既定模式进行检查：

- [ ] 代码是否遵循项目约定？
- [ ] 命名是否清晰且一致？
- [ ] 代码是否足够 DRY（Don't Repeat Yourself）？
- [ ] 抽象层次是否恰当？
- [ ] 是否有应提取为工具函数的重复逻辑？

### 5. 报告结果

总结发现：

- **严重问题**：必须修复后才能合并
- **建议改进**：应当考虑但不阻塞合并
- **积极发现**：做得好的地方，值得保留
