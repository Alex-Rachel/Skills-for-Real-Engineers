# 请求重构计划

当用户要求重构时，不要直接动手——先制定计划。

## 为什么先计划

重构没有计划是危险的：

- 可能引入难以追踪的 bug
- 可能遗漏必要的级联变更
- 可能做无关的修改，扩大变更范围
- 难以中途验证正确性

## 工作流程

### 1. 理解重构目标

提问以明确意图：

- "你希望通过重构达成什么目标？"
- "当前代码的什么问题让你想要重构？"
- "有没有不能改变的外部接口？"

### 2. 分析当前代码

审查待重构的代码：

- 识别所有依赖关系
- 找出哪些地方会受变更影响
- 理解当前行为（包括隐含的）
- 识别现有的测试覆盖

### 3. 制定重构计划

产出分步骤的计划，每一步都是安全且可验证的：

1. **准备阶段**：添加测试以锁定现有行为
2. **重构步骤**：每步只做一个小改动
3. **验证步骤**：每步之后运行测试确认行为不变
4. **收尾阶段**：清理临时代码、更新文档

### 4. 呈现计划

以清晰的格式呈现计划：

```
重构目标：[目标]

步骤：
1. [第一步] - 原因：[为什么]
2. [第二步] - 原因：[为什么]
...

风险：
- [潜在风险及缓解措施]

预计影响范围：
- [受影响的文件/模块]
```

### 5. 获得确认

在执行之前请求用户确认：

- "这个计划看起来如何？"
- "有需要调整的步骤吗？"
- "准备好开始执行了吗？"
