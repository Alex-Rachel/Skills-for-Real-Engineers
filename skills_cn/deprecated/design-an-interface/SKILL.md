# 设计接口

源自《软件设计哲学》中的"设计两次"原则：你的第一个想法往往不是最好的。生成多种截然不同的设计方案，然后进行比较。

## 工作流程

### 1. 收集需求

在设计之前，先了解：

- [ ] 这个模块解决什么问题？
- [ ] 谁是调用方？（其他模块、外部用户、测试）
- [ ] 关键操作有哪些？
- [ ] 有什么约束？（性能、兼容性、既有模式）
- [ ] 哪些应该隐藏在内部，哪些应该暴露？

提问："这个模块需要做什么？谁会使用它？"

### 2. 生成设计方案（并行子代理）

使用 Task 工具同时启动 3 个以上的子代理。每个子代理必须产出一种**截然不同**的方案。

```
每个子代理的提示模板：

为以下模块设计接口：[模块描述]

需求：[收集到的需求]

本设计的约束：[为每个代理分配不同的约束]
- 代理 1："最小化方法数量——目标是最多 1-3 个方法"
- 代理 2："最大化灵活性——支持尽可能多的用例"
- 代理 3："针对最常见场景进行优化"
- 代理 4："借鉴[特定范式/库]的思路"

输出格式：
1. 接口签名（类型/方法）
2. 使用示例（调用方如何使用）
3. 本设计在内部隐藏了什么
4. 该方案的权衡取舍
```

### 3. 展示设计方案

逐一展示每个设计：

1. **接口签名** —— 类型、方法、参数
2. **使用示例** —— 调用方在实际中如何使用
3. **隐藏内容** —— 保留在内部的复杂度

按顺序展示设计，让用户在比较之前充分理解每种方案。

### 4. 比较设计方案

展示完所有设计后，从以下维度进行比较：

- **接口简洁性**：方法更少、参数更简单
- **通用 vs 专用**：灵活性与专注性的权衡
- **实现效率**：接口形态是否允许高效的内部实现？
- **深度**：小接口隐藏大量复杂度（好）vs 大接口搭配薄实现（差）
- **正确使用的难度** vs **误用的难度**

用文字讨论权衡取舍，而非表格。突出设计方案之间分歧最大的地方。

### 5. 综合

最佳设计往往融合了多种方案的洞见。提问：

- "哪种设计最符合你的主要用例？"
- "其他设计中是否有值得借鉴的元素？"

## 评估标准

源自《软件设计哲学》：

**接口简洁性**：方法更少、参数更简单 = 更容易学习和正确使用。

**通用性**：无需修改即可应对未来用例。但要警惕过度泛化。

**实现效率**：接口形态是否允许高效实现？还是迫使内部实现变得别扭？

**深度**：小接口隐藏大量复杂度 = 深模块（好）。大接口搭配薄实现 = 浅模块（应避免）。

## 反模式

- 不要让子代理产出相似的设计——必须确保方案之间有根本性差异
- 不要跳过比较环节——价值在于对比
- 不要进行实现——这纯粹是关于接口形态的讨论
- 不要以实现工作量作为评估依据
