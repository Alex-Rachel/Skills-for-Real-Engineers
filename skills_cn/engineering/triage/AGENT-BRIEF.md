# 编写 Agent 简报

Agent 简报是一个结构化评论，当 issue 移到 `ready-for-agent` 时发布在 GitHub issue 上。它是 AFK agent 工作的权威规格。原始 issue 正文和讨论是上下文——agent 简报是契约。

## 原则

### 耐久性优于精确性

issue 可能在 `ready-for-agent` 中待上几天或几周。在此期间代码库会变化。编写简报时要使其即使文件被重命名、移动或重构后仍然有用。

- **要** 描述接口、类型和行为契约
- **要** 命名 agent 应该查找或修改的具体类型、函数签名或配置形状
- **不要** 引用文件路径——它们会过时
- **不要** 引用行号
- **不要** 假设当前实现结构会保持不变

### 行为性，而非过程性

描述系统**应该做什么**，而非**如何实现**。Agent 会重新探索代码库并做出自己的实现决策。

- **好的：** "`SkillConfig` 类型应该接受一个可选的 `schedule` 字段，类型为 `CronExpression`"
- **坏的：** "打开 src/types/skill.ts 在第 42 行添加 schedule 字段"
- **好的：** "当用户不带参数运行 `/triage` 时，应该看到需要关注的 issue 摘要"
- **坏的：** "在主处理函数中添加 switch 语句"

### 完整的验收标准

Agent 需要知道什么时候算完成。每个 agent 简报必须有具体、可测试的验收标准。每个标准应该可以独立验证。

- **好的：** "运行 `gh issue list --label needs-triage` 返回经过初始分类的 issue"
- **坏的：** "分流应该正常工作"

### 明确的范围边界

说明什么不在范围内。这防止 agent 镀金或对相邻功能做出假设。

## 模板

```markdown
## Agent 简报

**类别：** bug / enhancement
**摘要：** 一行描述需要做什么

**当前行为：**
描述现在发生了什么。对于 bug，这是异常行为。
对于增强，这是功能构建于其上的现状。

**期望行为：**
描述 agent 工作完成后应该发生什么。
对边缘情况和错误条件要具体。

**关键接口：**
- `TypeName` — 什么需要改变以及为什么
- `functionName()` 返回类型 — 当前返回什么 vs 应该返回什么
- 配置形状 — 需要的任何新配置选项

**验收标准：**
- [ ] 具体的、可测试的标准 1
- [ ] 具体的、可测试的标准 2
- [ ] 具体的、可测试的标准 3

**不在范围内：**
- 在此 issue 中不应该被改变或处理的东西
- 看似相关但实际独立的相邻功能
```

## 示例

### 好的 agent 简报（bug）

```markdown
## Agent 简报

**类别：** bug
**摘要：** 技能描述截断在单词中间断开，产生破损输出

**当前行为：**
当技能描述超过 1024 个字符时，会在恰好 1024 个字符处截断，
不考虑单词边界。这产生在单词中间结束的描述
（例如 "Use when the user wants to confi"）。

**期望行为：**
截断应该在 1024 个字符之前的最后一个单词边界处断开
并附加 "..." 表示截断。

**关键接口：**
- `SkillMetadata` 类型的 `description` 字段 — 不需要类型变更，
  但填充它的验证/处理逻辑需要尊重单词边界
- 任何读取 SKILL.md frontmatter 并提取描述的函数

**验收标准：**
- [ ] 1024 字符以下的描述不变
- [ ] 超过 1024 字符的描述在 1024 字符之前的最后一个单词边界处截断
- [ ] 截断的描述以 "..." 结尾
- [ ] 包含 "..." 的总长度不超过 1024 字符

**不在范围内：**
- 改变 1024 字符限制本身
- 多行描述支持
```

### 好的 agent 简报（enhancement）

```markdown
## Agent 简报

**类别：** enhancement
**摘要：** 添加 `.out-of-scope/` 目录支持，用于跟踪被拒绝的功能请求

**当前行为：**
当功能请求被拒绝时，issue 用 `wontfix` 标签关闭
并附评论。没有持久的决策记录或推理。
未来类似的请求需要维护者回忆或搜索
之前的讨论。

**期望行为：**
被拒绝的功能请求应该记录在 `.out-of-scope/<concept>.md`
文件中，捕获决策、推理和所有请求该功能的 issue 链接。
在分流新 issue 时，应该检查这些文件以寻找匹配。

**关键接口：**
- `.out-of-scope/` 中的 Markdown 文件格式 — 每个文件应有
  `# Concept Name` 标题、`**Decision:**` 行、`**Reason:**` 行
  和带 issue 链接的 `**Prior requests:**` 列表
- 分流工作流应早期读取所有 `.out-of-scope/*.md` 文件
  并通过概念相似性将传入的 issue 与之匹配

**验收标准：**
- [ ] 将功能关闭为 wontfix 时创建/更新 `.out-of-scope/` 中的文件
- [ ] 文件包含决策、推理和被关闭 issue 的链接
- [ ] 如果匹配的 `.out-of-scope/` 文件已存在，新 issue
      被追加到其"Prior requests"列表，而非创建重复
- [ ] 分流时，当新 issue 匹配先前拒绝时，检查并呈现
      现有的 `.out-of-scope/` 文件

**不在范围内：**
- 自动匹配（人确认匹配）
- 重新打开先前被拒绝的功能
- Bug 报告（只有增强拒绝才进入 `.out-of-scope/`）
```

### 坏的 agent 简报

```markdown
## Agent 简报

**摘要：** 修复分流 bug

**做什么：**
分流的东西坏了。看看主文件然后修一下。
第 150 行附近的函数有问题。

**要改的文件：**
- src/triage/handler.ts（第 150 行）
- src/types.ts（第 42 行）
```

这是坏的因为：
- 没有类别
- 描述模糊（"分流的东西坏了"）
- 引用会过时的文件路径和行号
- 没有验收标准
- 没有范围边界
- 没有描述当前行为 vs 期望行为
