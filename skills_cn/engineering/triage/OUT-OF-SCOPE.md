# 范围外

如何处理超出项目范围的功能请求。

## 原则

- **记录拒绝，不要只是关闭。** 每个被拒绝的功能请求都应有持久记录解释为什么。这防止相同的请求被反复提出。
- **记录是给未来的分流者看的。** 当新 issue 进来时，分流者应该能快速检查是否有先前拒绝的相似请求。

## 流程

### 1. 识别范围外请求

当功能请求超出项目范围时：
- 不符合项目目标
- 会引入不可接受的复杂性
- 更好地由另一个工具/项目解决
- 维护者已明确拒绝

### 2. 记录决策

在 `.out-of-scope/` 目录中创建 Markdown 文件：

```markdown
# 概念名称

**决策：** 拒绝 / 推迟

**原因：** 为什么此功能不在范围内。具体说明——
"这会需要为单个用例维护一个完整的渲染管道"
比"太复杂"更有用。

**先前请求：**
- #42 — 2024-01-15 的原始请求
- #87 — 2024-02-20 的重复请求
```

文件名应该是概念名称的 kebab-case：`.out-of-scope/visual-wysiwyg.md`。

### 3. 关闭 issue

关闭 issue 并标记 `wontfix`。添加评论：

> 此功能请求超出 [项目名称] 的范围。原因记录在 `.out-of-scope/<concept>.md` 中。

### 4. 分流时参考

在分流新 issue 时，检查 `.out-of-scope/` 目录。如果新 issue 匹配先前拒绝的请求：

1. 关闭为重复
2. 引用 `.out-of-scope/` 文件
3. 如果推理可能已改变，可选择询问维护者

## 维护

定期审查 `.out-of-scope/` 文件。如果项目目标或约束已改变，先前拒绝的功能可能变得可行。更新文件以反映当前推理。
