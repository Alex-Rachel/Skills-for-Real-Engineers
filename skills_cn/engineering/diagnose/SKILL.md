# Diagnose

一门针对疑难 bug 的纪律。跳过某个阶段必须有明确的理由。

在探索代码库时，使用项目的领域术语表来建立相关模块的清晰心智模型，并检查你涉及区域的 ADR（架构决策记录）。

## 阶段 1 — 构建反馈循环

**这就是核心技能。** 其他一切都是机械操作。如果你能为 bug 建立一个快速、确定性、可由 agent 运行的通过/失败信号，你就能找到原因——二分法、假设检验和插桩都只是消费这个信号。如果没有这样的信号，再怎么盯着代码看也救不了你。

在这里投入不成比例的精力。**要激进。要有创意。绝不放弃。**

### 构建方式 — 大致按以下顺序尝试

1. **失败的测试** — 在任何能触及 bug 的接缝处：单元、集成、端到端。
2. **Curl / HTTP 脚本** — 对运行中的开发服务器发起请求。
3. **CLI 调用** — 使用固定输入，将 stdout 与已知正确的快照进行 diff。
4. **无头浏览器脚本**（Playwright / Puppeteer）— 驱动 UI，对 DOM/控制台/网络进行断言。
5. **回放捕获的追踪。** 将真实的网络请求/载荷/事件日志保存到磁盘；在隔离环境中通过代码路径回放。
6. **一次性测试桩。** 启动系统的最小子集（一个服务，mock 掉依赖），用单个函数调用执行 bug 代码路径。
7. **属性 / 模糊测试循环。** 如果 bug 是"有时输出错误"，运行 1000 个随机输入并寻找失败模式。
8. **二分法测试桩。** 如果 bug 出现在两个已知状态（提交、数据集、版本）之间，自动化"启动状态 X，检查，重复"，这样就可以用 `git bisect run` 来执行。
9. **差分循环。** 将相同输入分别通过旧版本和新版本（或两种配置）运行，diff 输出结果。
10. **HITL bash 脚本。** 最后手段。如果必须由人来点击，用 `scripts/hitl-loop.template.sh` 来驱动他们，使循环仍然结构化。捕获的输出会反馈给你。

构建正确的反馈循环，bug 就修复了 90%。

### 对循环本身进行迭代

把循环当作产品来对待。一旦你有了一个循环，就问：

- 能让它更快吗？（缓存设置、跳过无关初始化、缩小测试范围。）
- 能让信号更清晰吗？（断言具体症状，而不是"没崩溃"。）
- 能让它更确定性吗？（固定时间、固定随机种子、隔离文件系统、冻结网络。）

一个 30 秒的脆弱循环几乎不比没有循环好。一个 2 秒的确定性循环是调试的超能力。

### 非确定性 bug

目标不是一个干净的复现，而是**更高的复现率**。循环触发 100 次、并行化、增加压力、缩小时间窗口、注入 sleep。一个 50% 概率出现的 bug 是可以调试的；1% 则不行——持续提高复现率，直到可以调试为止。

### 当你确实无法构建循环时

停下来，明确说明。列出你尝试过的方法。向用户请求：(a) 访问能复现问题的环境，(b) 捕获的产物（HAR 文件、日志转储、core dump、带时间戳的屏幕录制），或 (c) 添加临时生产环境插桩的权限。在没有循环的情况下**不要**进入假设阶段。

在你拥有一个你信任的循环之前，不要进入阶段 2。

## 阶段 2 — 复现

运行循环。观察 bug 出现。

确认：

- [ ] 循环产生的是**用户**描述的失败模式——不是碰巧在附近的另一种失败。错误的 bug = 错误的修复。
- [ ] 失败在多次运行中可复现（或者，对于非确定性 bug，以足够高的比率复现以供调试）。
- [ ] 你已捕获确切的症状（错误消息、错误输出、慢速计时），以便后续阶段可以验证修复确实解决了问题。

在复现 bug 之前不要继续。

## 阶段 3 — 假设

在测试任何假设之前，生成 **3–5 个排好序的假设**。单假设生成会锚定在第一个看似合理的想法上。

每个假设必须是**可证伪的**：陈述它做出的预测。

> 格式："如果 <X> 是原因，那么 <改变 Y> 会使 bug 消失 / <改变 Z> 会使它更严重。"

如果你无法陈述预测，那这个假设只是一种感觉——丢弃或锐化它。

**在测试之前将排序列表展示给用户。** 他们通常有领域知识可以立即重排（"我们刚部署了 #3 的变更"），或者知道已经排除的假设。低成本的检查点，巨大的时间节省。如果用户不在，不要阻塞——按你的排序继续。

## 阶段 4 — 插桩

每个探针必须映射到阶段 3 中的特定预测。**一次只改变一个变量。**

工具偏好：

1. **调试器 / REPL 检查** — 如果环境支持。一个断点胜过十条日志。
2. **目标日志** — 在区分假设的边界处。
3. 永远不要"记录一切然后 grep"。

**为每条调试日志打标签**，使用唯一前缀，例如 `[DEBUG-a4f2]`。最终清理只需一次 grep。未打标签的日志会残留；打标签的日志会被清除。

**性能分支。** 对于性能回归，日志通常是错误的。取而代之：建立基线测量（计时工具、`performance.now()`、性能分析器、查询计划），然后二分。先测量，后修复。

## 阶段 5 — 修复 + 回归测试

在修复**之前**编写回归测试——但仅当存在**正确的接缝**时。

正确的接缝是指测试在该处能演练**真实的 bug 模式**，正如它在调用点处发生的那样。如果唯一可用的接缝太浅（bug 需要多个调用者但只有单调用者测试、单元测试无法复现触发 bug 的调用链），那里的回归测试会给出虚假的信心。

**如果不存在正确的接缝，这本身就是发现。** 记录下来。代码库架构阻止了 bug 被锁定。将此标记给下一阶段。

如果存在正确的接缝：

1. 将最小化复现转换为该接缝处的失败测试。
2. 观察它失败。
3. 应用修复。
4. 观察它通过。
5. 针对原始（未最小化的）场景重新运行阶段 1 的反馈循环。

## 阶段 6 — 清理 + 复盘

在宣布完成之前必须完成：

- [ ] 原始复现不再复现（重新运行阶段 1 循环）
- [ ] 回归测试通过（或接缝缺失已记录）
- [ ] 所有 `[DEBUG-...]` 插桩已移除（grep 该前缀）
- [ ] 一次性原型已删除（或移至明确标记的调试位置）
- [ ] 被证实为正确的假设已写入 commit / PR 消息——让下一位调试者学到

**然后问：什么能防止这个 bug？** 如果答案涉及架构变更（没有好的测试接缝、纠缠的调用者、隐式耦合），将具体情况移交给 `/improve-codebase-architecture` 技能。在修复完成**之后**提出建议，而不是之前——你现在拥有的信息比开始时更多。
