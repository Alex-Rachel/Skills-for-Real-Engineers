# 深模块

来自 John Ousterhout 的《软件设计的哲学》。

## 核心思想

模块的**深度**由其接口的简单性与实现的复杂性之比决定。

- **深模块**：简单接口，复杂实现。接口提供的杠杆高——少量表面积控制大量行为。
- **浅模块**：接口几乎与实现一样复杂。接口提供的杠杆低——你通过接口没有获得多少抽象。

```
深模块：
┌──────────────────────┐
│  简单接口            │  ← 3 个方法
│ ┌──────────────────┐ │
│ │                  │ │
│ │  复杂实现        │ │  ← 500 行逻辑
│ │                  │ │
│ └──────────────────┘ │
└──────────────────────┘

浅模块：
┌──────────────────────┐
│  复杂接口            │  ← 15 个方法
│ ┌──────────────────┐ │
│ │                  │ │
│ │  简单实现        │ │  ← 5 行逻辑
│ │                  │ │
│ └──────────────────┘ │
└──────────────────────┘
```

## 为什么深度重要

深模块通过两种方式降低系统复杂性：

1. **接口隐藏了实现细节。** 调用者不需要知道模块内部如何工作。这减少了调用者必须理解的概念量。
2. **变更局部化。** 当实现变更时，只有模块本身需要修改。接口提供了稳定的契约。

浅模块做反两面：它们暴露实现细节（调用者必须理解内部），且当实现变更时，调用者和模块都可能需要修改。

## 识别浅模块

常见信号：

- **传递模块** — 只转发调用，不增加价值。`a.foo()` 调用 `b.foo()`。
- **包装模块** — 添加一层间接但无逻辑。`new Validator(input).validate()` 只调用 `input.validate()`。
- **配置模块** — 只组装其他模块，不含领域逻辑。整个文件都是 DI 配线。
- **泄漏模块** — 将内部细节暴露给调用者。调用者必须了解模块的内部状态才能正确使用它。

## 加深模块

加深模块的策略：

1. **合并相关模块。** 如果两个模块总是协同使用，将它们合并为一个接口更简单的模块。
2. **将逻辑推到接口后面。** 如果调用者做的工作本可以由模块完成，将逻辑移入模块。
3. **消除泄漏。** 如果模块暴露了调用者不需要的细节，封装它们。
4. **提升抽象级别。** 如果接口太低级（暴露文件路径、数据库查询），提升到领域级别。

## 与 TDD 的关系

TDD 鼓励深模块，因为：

- 测试通过公共接口编写，迫使接口成为主要抽象
- 重构步骤主动寻找合并和简化的机会
- 测试先的方法自然产生行为级接口而非实现级接口
